# 家政服务系统 — 需求分析文档

## 课题名称：基于Spring Boot+Vue的家政服务系统的设计与实现

---

## 一、引言

### 1.1 编写目的

本文档对家政服务系统进行详细的需求分析，明确系统的功能需求、非功能需求、用户角色及业务流程，为后续的系统设计和开发提供依据。

### 1.2 系统概述

本系统是一个基于B/S架构的家政服务管理平台，采用Spring Boot + Vue前后端分离技术实现。系统面向三类用户：**普通用户（雇主）**、**服务人员（家政员）** 和 **系统管理员**，提供家政服务的在线预约、订单管理、评价反馈等功能。

### 1.3 术语定义

| 术语 | 说明 |
|------|------|
| 用户/雇主 | 需要家政服务的客户 |
| 服务人员/阿姨 | 提供家政服务的工作人员 |
| 管理员 | 系统后台管理人员 |
| 订单 | 用户预约家政服务生成的记录 |
| 服务类目 | 家政服务的分类，如保洁、月嫂、保姆等 |

---

## 二、用户角色分析

系统涉及三类用户角色：

### 2.1 普通用户（雇主）

- 需要查找和预约家政服务
- 需要管理自己的订单
- 需要对服务进行评价
- 需要管理个人信息和收货地址

### 2.2 服务人员（家政员）

- 需要展示个人信息和技能
- 需要接收和管理派单
- 需要查看自己的收入和评价
- 需要管理个人工作状态

### 2.3 系统管理员

- 需要管理用户和服务人员信息
- 需要管理服务类目
- 需要管理订单和处理异常
- 需要查看系统运营数据

## 四、用例描述

### 4.1 用户预约服务用例

```
用例名称：预约家政服务
参与者：  普通用户
前置条件：用户已登录系统
基本流程：
  1. 用户在首页浏览服务类目
  2. 用户点击进入某个服务类目，查看服务列表
  3. 用户选择某个服务，查看服务详情
  4. 用户点击"立即预约"按钮
  5. 系统显示预约页面，用户选择服务时间
  6. 用户选择或填写服务地址
  7. 用户确认订单信息，点击"提交订单"
  8. 系统生成订单，显示预约成功页面
  9. 系统通知管理员有新订单待处理
替代流程：
  5a. 用户未登录 → 系统跳转到登录页面
  6a. 用户没有保存地址 → 系统提示用户添加地址
后置条件：系统中生成一条待接单状态的订单
```

### 4.2 管理员派单用例

```
用例名称：管理员派单
参与者：  系统管理员
前置条件：管理员已登录后台系统，存在待派单订单
基本流程：
  1. 管理员进入订单管理页面
  2. 管理员筛选"待派单"状态的订单
  3. 管理员点击某个订单，查看订单详情
  4. 管理员点击"派单"按钮
  5. 系统显示符合条件的服务人员列表
  6. 管理员选择一个服务人员，确认派单
  7. 系统更新订单状态为"已派单"
  8. 系统通知服务人员有新订单
替代流程：
  5a. 没有符合条件的服务人员 → 管理员取消派单，稍后处理
后置条件：订单状态更新为"已派单"，服务人员收到通知
```

### 4.3 用户评价用例

```
用例名称：用户评价服务
参与者：  普通用户
前置条件：用户已登录，存在已完成的订单且未评价
基本流程：
  1. 用户进入"我的订单"页面
  2. 用户找到已完成的订单
  3. 用户点击"评价"按钮
  4. 系统显示评价页面
  5. 用户选择星级评分
  6. 用户填写评价内容
  7. 用户点击"提交评价"
  8. 系统保存评价信息
  9. 系统更新服务人员的综合评分
后置条件：订单标记为"已评价"，服务人员评分更新
```

## 五、业务流程分析

### 5.1 订单核心业务流程

```
用户浏览服务 → 选择服务 → 提交预约订单
                                    │
                                    ▼
                            订单状态：待派单
                                    │
                                    ▼
                          管理员审核并派单
                                    │
                                    ▼
                            订单状态：已派单
                                    │
                                    ▼
                          服务人员接单/拒单
                         /               \
                       接单              拒单
                        │                 │
                        ▼                 ▼
                  订单状态：进行中    重新派单
                        │
                        ▼
                  服务人员完成服务
                        │
                        ▼
                  订单状态：待评价
                        │
                        ▼
                    用户评价
                        │
                        ▼
                  订单状态：已完成
```

### 5.2 用户注册流程

```
用户打开注册页面 → 输入手机号 → 输入密码 → 确认密码
       │
       ▼
  系统验证输入格式
       │
    格式正确？──否──→ 提示错误信息，返回修改
       │
      是
       │
       ▼
  检查手机号是否已注册
       │
    未注册？──否──→ 提示手机号已注册
       │
      是
       │
       ▼
  保存用户信息到数据库
       │
       ▼
  注册成功，跳转到登录页面
```

## 六、非功能需求

### 6.1 性能需求

| 指标 | 要求 |
|------|------|
| 页面响应时间 | 普通页面加载时间 ≤ 3秒 |
| 接口响应时间 | API接口平均响应时间 ≤ 500ms |
| 并发用户数 | 支持 ≥ 100个并发用户 |
| 数据库查询 | 单表查询时间 ≤ 200ms |

### 6.2 安全需求

| 需求 | 说明 |
|------|------|
| 身份认证 | 采用JWT Token进行用户身份验证 |
| 密码加密 | 用户密码使用BCrypt加密存储 |
| 权限控制 | 基于角色的访问控制（RBAC），不同角色只能访问对应功能 |
| SQL注入防护 | 使用MyBatis-Plus参数化查询，防止SQL注入 |
| XSS防护 | 前端对用户输入进行转义处理 |

### 6.3 可用性需求

| 需求 | 说明 |
|------|------|
| 界面友好 | 界面简洁美观，操作流程直观 |
| 响应式设计 | 适配主流浏览器（Chrome、Firefox、Edge） |
| 错误提示 | 操作失败时给出明确的错误提示信息 |
| 兼容性 | 支持1920×1080及以上分辨率 |

### 6.4 可维护性需求

| 需求 | 说明 |
|------|------|
| 代码规范 | 遵循Java和Vue编码规范 |
| 注释完整 | 关键方法和类添加注释说明 |
| 分层架构 | 代码按Controller-Service-Mapper分层组织 |
| 接口文档 | 使用Knife4j自动生成API文档 |

## 八、需求优先级总结

- 服务人员收入统计
